Lesson 1: Before the Cloud & Virtualization

Before the Cloud (B.C.)

▼

Before cloud computing, infrastructure was simple but inefficient:

  • One physical server ran one OS and typically one application
  • Underutilization was common — idle apps wasted hardware
  • Scaling required buying, shipping, installing, and cabling new servers
  • This process often took weeks
"If an application was not used, the computing resources of the server where it ran were wasted."
Key Problem

Physical servers were expensive, slow to provision, and often sat idle.

Virtualization Revolution

▼

Virtualization became mainstream in the early 2000s with technologies like XEN and KVM.

What is a Hypervisor?

Software that emulates hardware resources (CPU, RAM, disk, network) allowing multiple virtual machines to share one physical server.

Virtual Machine Components

▼
  • Its own Operating System
  • Virtual CPU
  • Virtual Memory
  • Virtual Disk
  • Virtual Network Interface Card (NIC)
  • Full isolation from other VMs

Benefits of Virtualization

▼
💰

Reduced hardware needs & lower costs

⚡

Lower power consumption

🚀

Faster provisioning (minutes vs weeks)

📦

Better resource utilization

Lesson 2: Cloud Computing & Service Models

What is Cloud Computing?

▼

Cloud computing = virtualization on demand

Cloud Characteristics

  • Hardware owned and managed by the provider
  • Customers pay only for what they use
  • Massive catalog of services: storage, networking, firewalls, GPUs, managed databases, etc.

Cloud Types

▼

☁️ Public Cloud

AWS, Azure, GCP

Multi-tenant

🏢 Private Cloud

VMware, Rackspace

Single-tenant

🔀 Hybrid Cloud

Mix of private + public

Flexible

Cloud Service Delivery Models

▼

IaaS

Infrastructure as a Service

  • Most flexible option
  • You manage: OS, updates, apps
  • Examples: EC2, EBS, Compute Engine

PaaS

Platform as a Service

  • Focus on apps; provider handles OS + runtime
  • Examples: Heroku, Google App Engine

SaaS

Software as a Service

  • Fully managed applications
  • Examples: Gmail, Slack

FaaS

Function as a Service / Serverless

  • Write functions triggered by events
  • No server management
  • Examples: AWS Lambda, Azure Functions

Shared Responsibility Model

▼

Defines what the cloud provider vs. customer must secure or maintain.

Provider Manages Customer Manages
Physical security Varies by model
Networking infrastructure Most in IaaS
Power & cooling Least in SaaS
Hardware maintenance Data & access control

Lesson 3: Cloud-Native & Containers

What is Cloud-Native?

▼

Cloud-native is an architectural approach emphasizing:

🔄 Resiliency
📈 Scalability
🤖 Automation
👁️ Observability
⚡ Dynamic Infrastructure
Important

Not tied to public cloud — Kubernetes on-prem is still cloud-native!

What are Containers?

▼

Containers are lightweight OS-level virtualization

Container Characteristics

▼
  • Share one host OS kernel
  • Each container has isolated:
    • Libraries & Dependencies
    • Processes
    • Networking
    • Storage

Container Advantages

▼
⚡
Extremely fast startup

Seconds vs minutes for VMs

🪶
Lightweight

Alpine images ~5-10 MB

📦
Highly portable

Run anywhere containers are supported

📊
Efficient resource usage

Many more containers than VMs per host

Container Images & Runtimes

▼

Container Images

Templates used to create containers.

"Every time we execute an image, a container is created."

Popular Container Runtimes

containerd CRI-O Docker Engine

Lesson 4: Monolithic vs Microservices

Monolithic Applications

▼
UI Business Logic Data Access Database

All components bundled together

Characteristics

  • ❌ All functionality bundled together
  • ❌ Hard to update individual components
  • ❌ Risky deployments (all or nothing)
  • ❌ Scaling is coarse and inefficient

Microservices Architecture

▼
Auth
Users
Orders
Payment
Inventory
Notifications

Independent services communicating over network

Each Service:

  • ✅ Has its own dedicated team
  • ✅ Deploys independently
  • ✅ Scales independently
  • ✅ Fails independently (fault isolation)

Lesson 5: Container Orchestration & Kubernetes

Why Container Orchestration?

▼

As microservices grew, container counts exploded. Managing hundreds or thousands of containers manually became impossible.

Popular Orchestrators

Kubernetes Industry Standard
OpenShift
Nomad
Docker Swarm
Apache Mesos

Kubernetes Origins

▼
🇬🇷

Name: "Kubernetes" means helmsman/pilot in Greek

⭐

Logo: Seven spokes (Project Seven → Star Trek reference)

🏢

Created: By Google in 2014

🤖

Inspired by: Google's internal system "Borg"

🎓

First CNCF graduated project

📊

One of the largest open-source projects on GitHub

Key Kubernetes Features

▼

📋 Declarative API

Define desired state, K8s makes it happen

🎯 Desired State Management

Continuously reconciles actual vs desired

📅 Scheduling

Automatically places containers on nodes

🔄 Rolling Updates

Zero-downtime deployments

📈 Autoscaling

Scale based on demand

💚 Self-Healing

Automatically restarts failed containers

Example: Self-Healing

If a node fails, Kubernetes automatically recreates missing containers on healthy nodes.
